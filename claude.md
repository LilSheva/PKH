# PKH Context Sniper (Google Colab Backend)

Утилита для умного поиска и извлечения контекста из сырых дампов чатов Google AI Studio (JSON) с использованием локальной векторной базы ChromaDB.
Код пишется в Codespaces, но будет исполняться в Google Colab (на примонтированном Google Drive).

## Основные правила (КРИТИЧНО)
1. **Игнорирование расширений файлов:** Файлы чатов приходят без расширения (формат JSON). Функция `is_chat_file(path)` читает первые 4 КБ файла. Если там есть `{` и паттерн `"chunkedPrompt"`, файл берется в работу. Игнорировать файлы только по явно бинарным сигнатурам, а не по имени.
2. **Парсинг JSON и Обработка размышлений:** Извлекать массив `chunkedPrompt.chunks`. 
   - 1 Блок (Chunk) = [Сообщение User] + [Сообщение Model].
   - ВСЕГДА игнорировать поля `"thoughtSignature"` (рекурсивно).
   - Обработка блоков `"isThought": true` регулируется `THOUGHT_MODE` (3 режима):
     * `OFF`: Полностью игнорировать размышления.
     * `ON`: Вклеивать весь текст размышлений перед финальным ответом.
     * `SMART`: Длинные размышления (> 1000 символов) разбиваются на абзацы, оставляются только те с cosine similarity > 0.4 к запросу User.
3. **Очистка от мусора:** `clean_text()` удаляет непрерывные буквенно-цифровые строки длиннее 100 символов (base64, токены, hex-дампы).
4. **Архитектура:** Модульный код. Удобный импорт в Jupyter Notebook. Точка входа — `ContextSniper` в `sniper.py`.

## Инкрементальный ингест и манифест файлов (КРИТИЧНО)

Архив постоянно пополняется: пользователь дописывает существующие чаты и создаёт новые. Ингест должен быть **идемпотентным** — повторный запуск на той же директории не плодит дублей и не пересканирует не-чаты.

### Манифест

Отдельный файл `MANIFEST_PATH = DB_DIR / "manifest.json"` хранит запись о **каждом просканированном файле** (включая не-чаты, чтобы не sniff'ить их повторно). Манифест общий для всех эмбеддинг-моделей — проверка «чат / не чат» от модели не зависит.

Схема записи:
```json
{
  "version": 1,
  "files": {
    "/abs/path/to/file": {
      "is_chat": true,
      "mtime": 1715212345.0,
      "size": 73919,
      "hash": "sha1 первых 64 КБ",
      "checked_at": "2026-05-09T12:00:00Z",
      "chat_id": "event_src.ip и event_src.asset",
      "system_instruction": null,
      "topics": [],
      "tags": []
    }
  }
}
```

Поля `topics` и `tags` сейчас не заполняются — заложены под будущую LLM-классификацию (теги вида `#программирование_python`, `#настройка_среды`). Поле `system_instruction` извлекается из JSON автоматически (см. раздел «Что НЕ ингестим»).

### Логика сканирования

1. Walk `CHATS_DIR`. Для каждого файла снять `(mtime, size)`.
2. Если запись в манифесте есть И `(mtime, size)` совпали → использовать кэшированный `is_chat`. Если `False` — скип, не sniff'ить. Если `True` и `hash` совпал — скип ингеста.
3. Если запись новая или `(mtime, size)` изменились → sniff'нуть (`is_chat_file`), пересчитать `hash`, обновить запись манифеста, поставить `checked_at = now()`.
4. Если `is_chat=True` и файл изменился — удалить из Chroma блоки этого `file_path` со старым `file_hash` и заново ингестить.
5. Сохранить манифест атомарно (write to tmp + rename) после прохода.

### Стабильный block_id и upsert

ID каждого `DialogBlock` детерминирован: `sha1(f"{chat_id}::{chunk_index}::{user_text[:200]}")`. Используем `collection.upsert(...)`. Дописанные сообщения добавляют новые блоки с новыми `chunk_index`; старые остаются на тех же ID.

### Дедупликация контента

- **Слой B (точный хеш контента, реализуем сразу).** Перед `upsert` каждого блока проверяем `collection.get(where={content_hash})`. Совпало — скип. `content_hash = sha1(нормализованный cleaned_content)`, нормализация = схлопывание whitespace.
- **Слой C (семантическая дедупликация, ЗАГЛУШКА).** Параметр `ENABLE_SEMANTIC_DEDUP = False`. Когда включён — перед добавлением блока делать `query` top-1 по существующей коллекции, при `cosine > SEMANTIC_DEDUP_THRESHOLD` (0.97) скипать. Сейчас оставить как `if ENABLE_SEMANTIC_DEDUP: pass  # TODO`.

### Что НЕ ингестим из JSON

- `thoughtSignature` (на любом уровне вложенности) — рекурсивно вычищается перед парсингом.
- `runSettings`, `pendingInputs` — служебные.
- Мультимодальные вложения чанков (`driveImage`, `driveDocument`, `inlineImage`, `inlineData.data`, `branchChildren`, `createTime`) — игнорируются. Парсер читает только `chunk["text"]` и `chunk["parts"][i]["text"]`, base64-данные не слурпает (важно для файлов десятки МБ).
- `systemInstruction` — НЕ попадает в Chroma как блок диалога, но **извлекается и сохраняется в манифесте** (поле `system_instruction`) для будущей LLM-классификации тегов. Пустой словарь `{}` (дефолт AI Studio) трактуется как `None`.

### Только JSON на входе

Файлы AI Studio приходят **в формате JSON, без расширения**. Никакого MD-парсинга на входе нет — MD это выходной формат (супер-промпт). Сканер опирается только на content sniffing (`is_chat_file`), расширения игнорируются.
